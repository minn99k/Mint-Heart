## 1. 시스템 아키텍처

### 1-1. 전체 구조

```
┌──────────────┐       WebSocket        ┌──────────────┐
│  Player 1    │◄─────── SignalR ──────►│  Player 2    │
│  Blazor WASM │                        │  Blazor WASM │
└──────┬───────┘                        └──────┬───────┘
       │                                       │
       └──────────────┬────────────────────────┘
                      │
                      ▼
            ┌───────────────────┐
            │  ASP.NET Core     │
            │                   │
            │  GameHub          │ ← SignalR 진입점 (이벤트 라우팅)
            │    │              │
            │  RoomService      │ ← 방 생성/관리
            │    │              │
            │  GameService      │ ← 상쇄, 점령, 승리 판정
            │    │              │
            │  GameLoopService  │ ← 틱 타이머, 생산, 전송, 이동
            │    │              │
            │  MapLoader        │ ← JSON 맵 로드
            │    │              │
            │  In-Memory Store  │ ← Dictionary<RoomId, GameRoom>
            └───────────────────┘
```

### 1-2. 서버/클라이언트 역할

| 서버 | 클라이언트 |
|------|------------|
| 게임 상태 관리 (진실의 원천) | UI 렌더링 (SVG) |
| 유닛 생산, 전송, 이동 계산 | 노드 클릭 이벤트 처리 |
| 상쇄, 점령, 승리 판정 | 이동 유닛 보간(lerp) 렌더링 |
| 입력 검증 (치트 방지) | 서버 상태 수신 → 화면 갱신 |
| 상태 브로드캐스트 | 전송 시작/중단 요청 전송 |

**핵심 원칙: 서버가 모든 게임 로직을 처리한다.** 클라이언트는 유저 입력을 서버에 보내고, 서버가 보내주는 결과를 받아서 화면에 그리기만 한다. 클라이언트의 데이터를 신뢰하지 않는다.

왜 이렇게 하냐면 — 클라이언트는 브라우저에서 돌아가니까 유저가 코드를 수정할 수 있다. "내 유닛 999개"로 조작하는 걸 막으려면 서버가 유일한 진실이어야 한다.

### 1-3. 통신 방식: SignalR

SignalR은 WebSocket을 감싸는 라이브러리다. 직접 WebSocket을 다루면 연결 관리, 재연결, 직렬화를 다 직접 해야 하는데, SignalR이 이걸 대신 해준다.

**SignalR 동작 방식:**

> [클라이언트]
>     │  Hub.SendAsync("CreateRoom", { mapId, nickname })
>     │────────────────────────────────►│
>                                       │  GameHub.CreateRoom() 실행
>                                       │  RoomService.CreateRoom() 호출
>     │  Clients.Caller.SendAsync("RoomCreated", roomInfo)
>     │◄────────────────────────────────│

- **Hub**: 서버에서 이벤트를 받는 클래스. 클라이언트가 "CreateRoom"을 보내면 GameHub의 CreateRoom 메서드가 호출된다.
- **Clients.Caller**: 요청을 보낸 클라이언트에게만 응답
- **Clients.Group(roomId)**: 같은 방에 있는 모든 클라이언트에게 브로드캐스트

## 2. Phase 0: 인프라 세팅 상세

> 기능명세서 F-00-1 ~ F-00-5에 대응. 코드 레벨에서 뭘 어떻게 세팅하는지.

### 2-1. 서버 Program.cs 세팅 (F-00-1)

Program.cs는 서버 앱의 진입점이다. 여기서 "어떤 서비스를 쓸지, 어떤 URL을 열지"를 설정한다.

**해야 할 것:**

> 1. SignalR 서비스 등록
>    → builder.Services.AddSignalR()
>    → "이 서버에서 SignalR을 사용하겠다"는 선언
>
> 2. CORS 설정
>    → 클라이언트(Blazor WASM)가 다른 포트에서 돌아가므로,
>      서버가 "이 출처에서 오는 요청은 허용한다"고 명시해야 함
>    → 안 하면 브라우저가 요청을 차단함 (보안 정책)
>
> 3. Hub 엔드포인트 매핑
>    → app.MapHub<GameHub>("/gamehub")
>    → "클라이언트가 /gamehub로 접속하면 GameHub 클래스가 처리한다"
>
> 4. 서비스 DI 등록 (F-00-5에서 추가)
>    → builder.Services.AddSingleton<RoomService>()
>    → builder.Services.AddSingleton<GameService>()
>    → builder.Services.AddSingleton<MapLoader>()
>    → builder.Services.AddHostedService<GameLoopService>()

**왜 Singleton인가:**
게임 상태는 서버 메모리에 있다. 요청마다 새 RoomService를 만들면 이전 방 데이터가 사라진다. Singleton으로 등록하면 서버가 살아있는 동안 하나의 인스턴스만 유지된다.

**왜 GameLoopService는 HostedService인가:**
게임 루프는 "누가 요청하지 않아도 매 틱마다 자동 실행"되어야 한다. HostedService(BackgroundService)는 서버 시작 시 자동으로 돌아가는 백그라운드 작업이다.

### 2-2. GameHub 구조 (F-00-2)

GameHub는 **클라이언트 요청의 진입점**이다. 직접 게임 로직을 처리하지 않고, 적절한 Service를 호출한 뒤 결과를 클라이언트에 돌려준다.

**GameHub의 역할:**

> 클라이언트 요청 → GameHub (라우터) → 적절한 Service → 결과 → 클라이언트 응답

- GameHub가 하는 것: 클라이언트 이벤트 수신, 요청을 Service로 위임, 결과를 클라이언트에 전송, 연결/끊김 이벤트 처리
- GameHub가 하지 않는 것: 방 코드 생성 로직(→ RoomService), 전투 계산(→ GameService), 유닛 이동(→ GameLoopService)

**왜 이렇게 나누냐면** — GameHub에 모든 로직을 넣으면 파일이 수천 줄이 되고, 두 사람이 동시에 수정할 때 충돌이 심해진다. Service로 분리하면 개발자A는 GameLoopService만, 개발자B는 GameService만 건드리면 된다.

### 2-3. 클라이언트 GameHubService (F-00-3)

클라이언트에서 서버 Hub와 통신하는 **래퍼 클래스**. Blazor 페이지들이 직접 HubConnection을 다루지 않고, 이 서비스를 통해 통신한다.

> Blazor 페이지 → GameHubService → SignalR HubConnection → 서버 GameHub

- GameHubService가 하는 것: HubConnection 생성 및 관리, 서버 이벤트 수신 등록, 서버에 이벤트 전송 메서드 제공, 연결 상태 관리
- Blazor 페이지가 하는 것: GameHubService를 주입받아서 메서드 호출, UI 렌더링

### 3-2. 방 관리

```csharp
public enum RoomStatus
{
    Waiting,    // 1명 접속, 상대 대기
    Starting,   // 2명 접속, 카운트다운 중
    InGame,     // 게임 진행 중
    Finished    // 게임 종료
}

public class GameRoom
{
    public string RoomId { get; set; }              // 숫자 6자리 ("482917")
    public string MapId { get; set; }               // "korea"
    public RoomStatus Status { get; set; }          // 방 상태
    public Dictionary<string, PlayerInfo> Players { get; set; }  // ConnectionId → 플레이어 정보
    public GameState GameState { get; set; }        // 게임 진행 상태
    public MapData MapData { get; set; }            // 맵 데이터 (노드 좌표 등)
    public DateTime CreatedAt { get; set; }         // 방 생성 시간
}

public class PlayerInfo
{
    public string ConnectionId { get; set; }        // SignalR이 부여하는 고유 ID
    public string PlayerId { get; set; }            // "player1" 또는 "player2"
    public string Nickname { get; set; }            // 유저가 입력한 닉네임
    public bool IsConnected { get; set; }           // 현재 연결 중인지
    public DateTime? DisconnectedAt { get; set; }   // 끊긴 시간 (재접속 30초 타이머용)
}
```

**서버 메모리에 방 목록을 어떻게 저장하는가:**

```csharp
// RoomService 내부
private readonly ConcurrentDictionary<string, GameRoom> _rooms = new();
```

ConcurrentDictionary를 쓰는 이유: 여러 플레이어가 동시에 방을 만들거나 입장할 수 있다. 일반 Dictionary는 동시 접근 시 데이터가 깨질 수 있는데, ConcurrentDictionary는 내부적으로 잠금(lock) 처리를 해줘서 안전하다.

**"deque로 관리"에 대해:**
기획서 메모에서 deque로 관리한다고 했는데, 서버 로직에서는 `int Units`로 충분하다. 왜냐면 유닛이 모두 동일해서 "어떤 유닛을 먼저 보낼지" 구분할 필요가 없다. 숫자만 관리하면 된다. 이동 중인 유닛은 MovingUnit 리스트에서 별도로 추적한다.

### 3-5. 전송 명령

기획서에서 "엣지 객체는 불필요"라고 했다. 대신 **"이 노드에서 저 노드로 계속 보내라"는 명령**만 존재한다.

```csharp
public class TransferOrder
{
    public string Id { get; set; }                  // 고유 ID (Guid 등)
    public string OwnerId { get; set; }             // 명령을 내린 플레이어 ("player1" / "player2")
    public string FromNodeId { get; set; }          // 출발 노드
    public string ToNodeId { get; set; }            // 도착 노드
    public DateTime CreatedAt { get; set; }         // 생성 시간 (라운드 로빈 순서 결정용)
}
```

**라운드 로빈이 뭔가:**
한 노드에서 여러 곳으로 전송할 때, 유닛을 어떤 순서로 보낼지 결정하는 방식.

> 서울에서 인천, 청주, 대구로 전송 중 (생성 순서: 인천 → 청주 → 대구)
> 매초 처리:
>   1번째 유닛 → 인천으로
>   2번째 유닛 → 청주로
>   3번째 유닛 → 대구로
>   4번째 유닛 → 인천으로 (다시 처음부터)
>   ...
>
> CreatedAt 순서대로 돌아가며 1유닛씩 보낸다.

### 3-6. 이동 유닛

```csharp
public class MovingUnit
{
    public string Id { get; set; }                  // 고유 ID
    public string OwnerId { get; set; }             // 소유 플레이어
    public string FromNodeId { get; set; }          // 출발 노드
    public string ToNodeId { get; set; }            // 도착 노드

    // 좌표 (맵의 픽셀 좌표)
    public double StartX { get; set; }              // 출발 x
    public double StartY { get; set; }              // 출발 y
    public double EndX { get; set; }                // 도착 x
    public double EndY { get; set; }                // 도착 y

    // 이동 계산
    public double Speed { get; set; }               // px/초 (맵의 unitSpeed)
    public double Distance { get; set; }            // 출발-도착 거리 (px)
    public DateTime DepartureTime { get; set; }     // 출발 시간
    public DateTime ArrivalTime { get; set; }       // 도착 예정 시간 = 출발 + (거리 ÷ 속도)

    public bool IsAlive { get; set; }               // true = 이동 중, false = 상쇄되어 소멸
}
```

**유닛 이동의 원리:**

> 서울(300, 150) → 인천(180, 170)
>
> 거리 = sqrt((300-180)² + (150-170)²) = sqrt(14400 + 400) = ~122px
> 속도 = 150px/초
> 도착 시간 = 출발시간 + (122 ÷ 150) = 출발 + 0.81초
>
> 매 틱마다 진행률 계산:
>   progress = (현재시간 - 출발시간) / (도착시간 - 출발시간)
>   currentX = 300 + (180 - 300) * progress
>   currentY = 150 + (170 - 150) * progress

## 4. API 설계 (SignalR 이벤트)

### 4-1. 이벤트 흐름 개요

> [클라이언트 → 서버] 유저가 행동하면 서버에 요청
> [서버 → 클라이언트] 서버가 처리 결과를 클라이언트에 알림

요청과 응답이 항상 1:1 대응이 아니다. 서버가 일방적으로 브로드캐스트하는 경우도 있다 (GameStateUpdated 등).

### 4-2. Client → Server

#### 로비/방

| 이벤트 | 파라미터 | 설명 | 성공 응답 | 실패 응답 |
|--------|----------|------|-----------|-----------|
| `CreateRoom` | mapId, nickname | 방 생성 | `RoomCreated` | `Error` |
| `JoinRoom` | roomId, nickname | 방 입장 | `RoomJoined` | `Error` |
| `LeaveRoom` | (없음) | 방 나가기 | `RoomLeft` | - |

#### 게임 중

| 이벤트 | 파라미터 | 설명 | 성공 응답 | 실패 응답 |
|--------|----------|------|-----------|-----------|
| `StartTransfer` | fromNodeId, toNodeId | 전송 시작 | `TransferStarted` | `Error` |
| `StopTransfer` | fromNodeId, toNodeId | 전송 중단 | `TransferStopped` | `Error` |

### 4-3. Server → Client

#### 로비/방

| 이벤트 | 파라미터 | 설명 | 누구에게 |
|--------|----------|------|----------|
| `RoomCreated` | RoomInfoDto | 방 생성 완료 | 요청자만 |
| `RoomJoined` | RoomInfoDto | 입장 성공 | 요청자만 |
| `PlayerJoined` | nickname, playerId | 상대 입장 알림 | 같은 방 전체 |
| `PlayerLeft` | playerId | 상대 퇴장 | 같은 방 전체 |
| `RoomLeft` | (없음) | 나가기 완료 | 요청자만 |
| `CountdownStarted` | seconds | 카운트다운 시작 | 같은 방 전체 |
| `GameStarted` | GameStateDto, myPlayerId | 게임 시작 + 초기 상태 | 같은 방 전체 (각자 myPlayerId는 다름) |

#### 게임 중

| 이벤트 | 파라미터 | 설명 | 누구에게 |
|--------|----------|------|----------|
| `GameStateUpdated` | GameStateDto | 주기적 상태 브로드캐스트 | 같은 방 전체 |
| `TransferStarted` | TransferOrderDto | 전송 시작 확인 | 같은 방 전체 |
| `TransferStopped` | fromNodeId, toNodeId | 전송 중단 확인 | 같은 방 전체 |
| `GameOver` | GameResultDto | 게임 종료 | 같은 방 전체 |
| `Error` | code, message | 에러 | 요청자만 |

#### 연결

| 이벤트 | 파라미터 | 설명 | 누구에게 |
|--------|----------|------|----------|
| `OpponentDisconnected` | (없음) | 상대 끊김, 30초 대기 시작 | 남은 플레이어 |
| `OpponentReconnected` | (없음) | 상대 재접속 | 남은 플레이어 |

### 4-4. "누구에게 보내는가"의 구현

```
Clients.Caller                → 요청을 보낸 사람에게만
Clients.Group(roomId)         → 같은 방 전체에게
Clients.OthersInGroup(roomId) → 같은 방에서 나를 제외한 상대에게
```

SignalR의 Group 기능을 사용한다. 방에 입장할 때 `Groups.AddToGroupAsync(connectionId, roomId)`로 그룹에 추가하면, 이후 `Clients.Group(roomId)`로 같은 방 사람들에게 한번에 브로드캐스트할 수 있다.

### 4-5. 에러 코드

| 코드 | 설명 | 발생 상황 |
|------|------|-----------|
| `ROOM_NOT_FOUND` | 존재하지 않는 방 | 잘못된 코드로 참가 시도 |
| `ROOM_FULL` | 이미 2명 | 꽉 찬 방에 참가 시도 |
| `NICKNAME_EMPTY` | 닉네임 미입력 | 빈 닉네임으로 요청 |
| `NICKNAME_TOO_LONG` | 12자 초과 | 긴 닉네임으로 요청 |
| `NOT_YOUR_NODE` | 내 노드가 아님 | 상대/중립 노드에서 전송 시도 |
| `SELF_TRANSFER` | 자기 자신에게 전송 | from == to |
| `DUPLICATE_TRANSFER` | 동일 쌍 중복 | 이미 전송 중인 쌍에 다시 전송 시도 |
| `MAX_TRANSFERS` | 노드당 3개 초과 | 4번째 전송 시도 |
| `TRANSFER_NOT_FOUND` | 없는 전송 중단 시도 | 존재하지 않는 전송을 중단 시도 |
| `GAME_NOT_ACTIVE` | 게임이 InGame이 아님 | 게임 시작 전이나 종료 후에 전송 시도 |

### 4-6. DTO (Data Transfer Object)

**DTO가 뭔가:** 서버 내부에서 쓰는 객체(GameRoom, GameState)를 그대로 클라이언트에 보내면 불필요한 정보(ConnectionId 등)가 노출된다. 클라이언트에 필요한 정보만 골라서 보내는 "전달용 객체"가 DTO다.

```csharp
// ===== 방 정보 =====
public class RoomInfoDto
{
    public string RoomId { get; set; }              // 방 코드
    public string MapId { get; set; }
    public string MapName { get; set; }
    public string Player1Nickname { get; set; }
    public string Player2Nickname { get; set; }     // null이면 아직 대기 중
}

// ===== 게임 상태 (브로드캐스트용) =====
public class GameStateDto
{
    public List<NodeStateDto> Nodes { get; set; }
    public List<TransferOrderDto> ActiveTransfers { get; set; }
    public List<MovingUnitDto> MovingUnits { get; set; }
    public int Player1TotalUnits { get; set; }      // P1 유닛 총합 (노드 + 이동 중)
    public int Player2TotalUnits { get; set; }
    public int Player1NodeCount { get; set; }       // P1 보유 노드 수
    public int Player2NodeCount { get; set; }
    public int ElapsedSeconds { get; set; }         // 경과 시간
}

public class NodeStateDto
{
    public string NodeId { get; set; }
    public string OwnerId { get; set; }             // "player1" / "player2" / null
    public int Units { get; set; }
    public int MaxUnits { get; set; }
}

public class TransferOrderDto
{
    public string Id { get; set; }
    public string OwnerId { get; set; }
    public string FromNodeId { get; set; }
    public string ToNodeId { get; set; }
}

public class MovingUnitDto
{
    public string Id { get; set; }
    public string OwnerId { get; set; }
    public double CurrentX { get; set; }            // 현재 x 좌표
    public double CurrentY { get; set; }            // 현재 y 좌표
    public double EndX { get; set; }                // 도착 x 좌표
    public double EndY { get; set; }                // 도착 y 좌표
    public double Progress { get; set; }            // 0.0 ~ 1.0 진행률
}

// ===== 게임 결과 =====
public class GameResultDto
{
    public string WinnerId { get; set; }            // null이면 무승부
    public string WinnerNickname { get; set; }
    public string WinReason { get; set; }           // "conquest" / "timeout" / "disconnect" / "draw"
    public int DurationSeconds { get; set; }
    public int Player1Nodes { get; set; }
    public int Player2Nodes { get; set; }
    public int Player1Units { get; set; }
    public int Player2Units { get; set; }
    public List<NodeStateDto> FinalNodes { get; set; }  // 최종 맵 상태 (결과 화면용)
}
```

## 5. 서비스 설계

### 5-1. 서비스 책임 요약

| 서비스 | 한 줄 요약 | 주요 메서드 |
|--------|-----------|-------------|
| **MapLoader** | JSON 맵 파일을 읽어서 메모리에 보관 | `LoadMap(mapId)`, `GetAvailableMaps()` |
| **RoomService** | 방 생성/입장/나가기/조회 | `CreateRoom()`, `JoinRoom()`, `LeaveRoom()`, `GetRoom()`, `GenerateRoomCode()` |
| **GameService** | 전송 명령 관리 + 상쇄/점령/승리 판정 | `StartTransfer()`, `StopTransfer()`, `ProcessArrival()`, `OnNodeCaptured()`, `CheckWinCondition()`, `InitializeGame()` |
| **GameLoopService** | 백그라운드 타이머, 매 틱마다 게임 상태 업데이트 | `Tick()` |

### 5-2. 서비스 간 호출 관계

> GameHub (진입점)
> - RoomService.CreateRoom() ← CreateRoom 이벤트 수신 시
> - RoomService.JoinRoom() ← JoinRoom 이벤트 수신 시
> - GameService.StartTransfer() ← StartTransfer 이벤트 수신 시
> - GameService.StopTransfer() ← StopTransfer 이벤트 수신 시
>
> GameLoopService (백그라운드, 자동 실행)
> - Tick() 매 틱마다:
>   - 유닛 생산 (직접 처리)
>   - 유닛 전송 (직접 처리)
>   - 유닛 이동 (직접 처리)
>   - GameService.ProcessArrival() ← 도착 상쇄/점령
>   - GameService.CheckWinCondition() ← 승리 판정
>   - Hub 브로드캐스트 (GameStateUpdated)

**GameLoopService가 GameService를 호출하는 이유:**
게임 루프(타이머)는 "매 틱마다 뭘 할지"만 관리하고, 실제 전투/점령 로직은 GameService에 위임한다. 이렇게 하면 개발자A(GameLoopService)와 개발자B(GameService)가 각자 파일만 수정하면 된다.

### 5-3. 게임 루프 상세 (GameLoopService.Tick)

타이머 간격: 100~200ms
- 1초보다 짧게 도는 이유: 유닛 이동 위치를 자주 업데이트해야 클라이언트에서 부드럽게 보임

**매 틱마다 실행 순서:**

1. **활성 게임방 순회** — Status == InGame인 방만 처리. Waiting, Starting, Finished인 방은 건너뜀

2. **[1초 간격] 유닛 자동 생산** — 마지막 생산으로부터 1초 경과했는지 체크. 각 플레이어 소유 노드: `Units += 1`. MaxUnits 초과하면 MaxUnits로 고정 (클램프). `ElapsedSeconds += 1`

3. **[1초 간격] 전송 명령 기반 유닛 발사** — 각 노드별로 ActiveTransfers를 CreatedAt 순서로 순회. 노드 유닛 > 0이면 Units -= 1, MovingUnit 새로 생성, 거리 계산, 도착 시간 계산. 노드 유닛 = 0이면 그 전송은 건너뜀 (차단). 유닛이 다시 생기면 다음 틱에서 자동 재개

4. **[매 틱] 이동 유닛 위치 업데이트** — 각 MovingUnit의 현재 위치를 시간 기반으로 계산. progress = (now - departureTime) / (arrivalTime - departureTime). progress >= 1.0이면 도착 처리 대상

5. **[매 틱] 이동 중 상쇄 판정** — IsAlive == true인 MovingUnit 중 서로 다른 소유자의 유닛 쌍을 비교. 두 유닛 간 거리 <= COLLISION_RADIUS (예: 10px)이면 양쪽 IsAlive = false (소멸)

6. **[매 틱] 노드 도착 처리** — ArrivalTime <= now이고 IsAlive == true인 유닛. GameService.ProcessArrival() 호출. 처리된 유닛은 MovingUnits에서 제거

7. **[매 틱] 승리 조건 체크** — GameService.CheckWinCondition() 호출. 한쪽 노드 0개 → 즉시 승리(conquest). ElapsedSeconds >= 300 → 노드 수 비교 → 유닛 수 비교(timeout/draw)

8. **[매 틱] 연결 끊김 체크** — DisconnectedAt != null인 플레이어가 있으면, 현재시간 - DisconnectedAt >= 30초 → 상대 몰수승(disconnect)

9. **[매 틱] 상태 브로드캐스트** — GameState → GameStateDto 변환. SignalR로 같은 방 양쪽 클라이언트에 전송

### 5-4. 이동 중 상쇄 판정 상세

```
모든 IsAlive == true인 MovingUnit 목록에서:

for each unitA in player1Units:
  for each unitB in player2Units:
    dx = unitA.currentX - unitB.currentX
    dy = unitA.currentY - unitB.currentY
    distance = sqrt(dx² + dy²)
    if distance <= COLLISION_RADIUS:
      unitA.IsAlive = false
      unitB.IsAlive = false
      break  // unitA는 이미 소멸, 다음 unitA로

같은 소유자끼리는 비교하지 않음 → 각자 기존 경로 유지
```

기획서에서 "서로 다른 경로에서 상대 유닛을 만나면 상쇄"라고 했으므로, **경로와 무관하게 거리 기반으로 판정**한다. 어떤 경로를 타든 가까이 있으면 상쇄.

**성능:** O(n²) 비교이지만, 12노드 5분 게임에서 동시 이동 유닛은 최대 50~100개 수준. 문제가 되면 이후 최적화.

### 5-5. 노드 점령 시 전송 자동 중단

```
GameService.OnNodeCaptured(nodeId, newOwnerId):

  1. 기존 소유자가 해당 노드에서 출발하는 전송을 전부 찾음
     oldTransfers = ActiveTransfers
       .Where(t => t.FromNodeId == nodeId && t.OwnerId != newOwnerId)

  2. 찾은 전송 전부 제거
     foreach (var t in oldTransfers):
       ActiveTransfers.Remove(t)

  3. 이미 이동 중인 유닛은 제거하지 않음
     → 목적지까지 계속 이동 (이미 출발한 유닛은 노드와 상관없이 이동하는 게 자연스러움)
```

**ProcessArrival 로직:**

- 아군 노드: `Units += 1` (MaxUnits 클램프)
- 적/중립 노드:
  - `Units > 0`: `Units -= 1` (상쇄)
  - `Units == 0`: 점령! → `OnNodeCaptured()` 호출. OwnerId 변경, Units = 1 (도착 유닛이 첫 유닛)

## 7. 성능 고려사항

### 이동 중 상쇄 O(n²) 문제

12노드, 5분 게임, 동시 이동 유닛 최대 50~100개 수준에서는 O(n²)이어도 문제없다. 100개 유닛이면 100×100 = 10,000번 비교인데, 현대 서버에서 이 정도는 1ms도 안 걸린다.

만약 유닛이 수백 개로 늘어나면 — player1 유닛과 player2 유닛을 분리해서 비교하거나, 공간을 격자로 나눠서 같은 격자 안의 유닛만 비교하는 최적화를 할 수 있다. MVP에서는 불필요.

### 상태 브로드캐스트 크기

매 틱(100~200ms)마다 GameStateDto 전체를 보낸다. 노드 12개 + 전송 명령 최대 36개(12노드 × 3개) + 이동 유닛 ~100개 정도면 JSON으로 수 KB 수준. WebSocket으로 충분히 감당 가능.

문제가 되면 — 변경된 부분만 보내는 diff 방식으로 전환할 수 있다. MVP에서는 전체 전송으로 시작.

### 타이머 정밀도

.NET의 Timer/Task.Delay는 정확한 간격을 보장하지 않는다. 100ms로 설정해도 실제로는 95~120ms 사이에서 흔들릴 수 있다.

대응: **모든 시간 계산은 절대 시간(DateTime.UtcNow) 기반으로 한다.** "틱이 몇 번 돌았는가"에 의존하지 않고, "현재 시간이 도착 예정 시간을 넘었는가"로 판정한다.

> 나쁜 예: tickCount * 100ms = 경과 시간 (틱 간격 오차가 누적됨)
> 좋은 예: DateTime.UtcNow - departureTime = 실제 경과 시간
