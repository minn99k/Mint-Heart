## 2. 게임 규칙 상세

### 2-1. 게임 구조 요약

- 맵 = 노드(도시) + 플레이어가 생성하는 엣지(연결선)
- 플레이어 = 2명
- 목표 = 상대의 모든 노드를 점령
- 수단 = 내 노드에서 목표 노드로 엣지를 연결하여 유닛을 자동 전송, 상대 유닛과 1:1 상쇄

### 2-2. 핵심 메카닉: 파이프라인 전송

이 게임의 핵심은 **"유닛을 한 번에 보내는 게 아니라, 파이프라인(엣지)을 연결하면 유닛이 자동으로 계속 흘러가는 것"**이다.

**기존 Sperm Wars 방식과의 차이:**

| | Sperm Wars (기존) | Capital Wars (본 게임) |
|---|---|---|
| 엣지 | 맵에 고정 (인접 노드끼리만) | 플레이어가 직접 생성 (아무 노드로) |
| 전송 | 클릭 시 1회 전송 (유닛의 50%) | 연결 유지 중 매초 자동 전송 |
| 전투 | 도착 시 유닛 수 비교 | 1:1 상쇄 (엣지 위에서도 발생) |
| 해제 | 없음 | 같은 노드 2개 다시 클릭 |

### 2-3. 노드 상태

모든 노드는 항상 아래 3가지 상태 중 하나에 속한다.

| 상태 | 조건 | 유닛 생산 |
|------|------|-----------|
| **Player1 소유** | OwnerId = Player1 | O (자동 생산) |
| **Player2 소유** | OwnerId = Player2 | O (자동 생산) |
| **중립** | OwnerId = null | X (유닛 고정, 생산 없음) |

- 게임 시작 시 각 플레이어에게 시작 노드 1개씩 배정
- 나머지 노드는 중립 상태 (초기 유닛 보유)
- 중립 노드는 유닛이 자동 증가하지 않음

### 2-4. 유닛 자동 생성

| 항목 | 값 |
|------|-----|
| 생성 주기 | 1초 |
| 생성량 | 점령 노드 1개당 1유닛/초 |
| 노드별 유닛 상한 | 노드마다 차등 (맵 데이터에서 정의) |

- 상한에 도달한 노드는 더 이상 유닛이 증가하지 않음
- 상한 차등 설정으로 거점 노드의 전략적 가치를 차별화

> **유닛 상한 차등 예시 (추후 밸런스 조정)**
> - 시작 노드(서울/부산): 상한 30
> - 중앙 거점(대전/대구): 상한 20
> - 일반 노드: 상한 15
> - 외곽 노드(제주/강릉): 상한 10

### 2-5. 엣지(연결) 시스템

이 게임에서 엣지는 맵에 고정된 것이 아니라, **플레이어가 직접 생성하고 해제하는 전략 요소**이다.

#### 엣지 생성

| 규칙 | 내용 |
|------|------|
| 생성 조건 | 내 노드 클릭 → 아무 노드(적/중립/아군) 클릭 |
| 연결 제한 | 자기 자신에게는 연결 불가 |
| 노드당 최대 연결 수 | **3개** (출발 기준) |
| 동일 엣지 중복 | 같은 출발-도착 쌍은 1개만 존재 가능 |

- 내가 점령한 노드에서만 엣지를 생성할 수 있음
- 상대 노드, 중립 노드, 아군 노드 어디든 도착지가 될 수 있음
- 연결 즉시 유닛 자동 전송이 시작됨

#### 엣지 해제

| 방법 | 설명 |
|------|------|
| 수동 해제 | 연결된 출발 노드 → 도착 노드를 다시 클릭 |
| 자동 해제 (점령 상실) | 출발 노드가 상대에게 점령당하면 해당 노드에서 나가는 모든 엣지 자동 해제 |

#### 노드당 최대 연결 3개의 밸런스 의도

- 유닛 생산 1유닛/초 vs 전송 소모 1~2유닛/초/연결 → 연결 3개면 유닛이 빠르게 소진
- 노드를 많이 점령할수록 여유가 생기지만, 연결을 많이 걸수록 방어가 약해짐
- "공격에 몇 개 할당하고, 지원에 몇 개 할당할지" 자원 배분 전략이 핵심

### 2-6. 유닛 전송

#### 자동 전송 규칙

| 항목 | 내용 |
|------|------|
| 전송량 | **1~2유닛/초/연결** (추후 테스트 후 확정) |
| 전송 조건 | 출발 노드의 유닛 > 0 |
| 전송 차단 | 출발 노드의 유닛이 0이면 자동으로 전송 정지, 유닛이 생기면 자동 재개 |
| 전송 우선순위 | 한 노드에서 여러 연결이 있을 때, 생성된 순서대로 1유닛씩 분배 (라운드 로빈) |

**전송 흐름 예시 (전송량 = 1유닛/초 기준):**

```
서울(유닛: 10) ──연결1──→ 대전(중립)
               ──연결2──→ 인천(중립)
               ──연결3──→ 대구(적)

매초 처리:
  서울 유닛 +1 (자동 생산)
  서울 유닛 -1 (연결1로 전송)
  서울 유닛 -1 (연결2로 전송)
  서울 유닛 -1 (연결3로 전송)
  → 순유닛 변화: -2/초

서울 유닛이 0이 되면:
  → 3개 연결 모두 전송 정지
  → 유닛이 1 이상 되면 연결1부터 순차 재개
```

#### 유닛 이동 속도

| 항목 | 내용 |
|------|------|
| 유닛 속도 | **고정값** (예: 150px/초, 추후 조정) |
| 도착 시간 | 두 노드 간 유클리드 거리(px) ÷ 유닛 속도 |
| 최소 이동 시간 | 0.5초 (아무리 가까워도) |

- 유닛은 일정한 속도로 이동하며, 먼 노드는 자연스럽게 오래 걸림
- 거리가 전략에 영향: 가까운 노드는 빠르게 점령 가능, 먼 노드 기습은 시간 리스크

**이동 시간 예시 (유닛 속도 = 150px/초 기준):**

| 경로 | 거리(px) | 이동 시간 |
|------|----------|-----------|
| 서울 → 인천 | ~122px | ~0.8초 → 최소 0.5초 적용 |
| 서울 → 청주 | ~112px | ~0.7초 → 최소 0.5초 적용 |
| 서울 → 대구 | ~283px | ~1.9초 |
| 서울 → 부산 | ~415px | ~2.8초 |
| 서울 → 제주 | ~443px | ~3.0초 |

### 2-7. 전투: 1:1 상쇄 시스템

이 게임에는 "전투 계산"이 없다. 모든 충돌은 **1:1 상쇄**로 처리된다.

#### 노드에서의 상쇄

내 유닛이 상대/중립 노드에 도착하면:

```
도착 유닛 vs 노드 방어 유닛 → 1:1 상쇄

예시: 내 유닛 1개 도착, 중립 노드 유닛 5개
  → 중립 노드 유닛 = 5 - 1 = 4
  → 점령 안 됨

매초 1유닛씩 계속 도착하면:
  5초 후 → 중립 노드 유닛 = 0
  6초 후 → 내 유닛 1개 도착 → 점령 성공 (노드 유닛 = 1, 소유권 전환)
```

#### 엣지 위에서의 상쇄

같은 경로에서 서로 반대 방향으로 이동하는 유닛이 만나면:

```
서울 → 대구: 3유닛 이동 중
대구 → 서울: 5유닛 이동 중

→ 3:3 상쇄 → 대구→서울 2유닛만 생존하여 계속 이동
```

#### 아군 노드로의 전송

- 내 유닛이 내 노드에 도착 → 유닛 합산 (상한 적용)

#### 점령 규칙

| 상황 | 결과 |
|------|------|
| 유닛 도착 시 노드 유닛 > 0 (상대/중립) | 1:1 상쇄, 소유권 변동 없음 |
| 유닛 도착 시 노드 유닛 = 0 (상대/중립) | 점령 성공, 소유권 전환, 도착 유닛이 노드 유닛이 됨 |
| 점령 성공 시 | 해당 노드에서 나가던 기존 소유자의 엣지 전부 자동 해제 |

### 2-8. 승리 조건

| 조건 | 설명 |
|------|------|
| **점령 승리** | 상대 플레이어의 소유 노드가 0개가 되면 즉시 승리 |
| **시간 제한** | 5분 경과 시 보유 노드 수 → 총 유닛 수 순으로 비교하여 승자 결정 |
| **동점 처리** | 노드 수, 유닛 수 모두 동일하면 무승부 |

**시간 제한(5분) 설정 이유:**
- 양쪽이 팽팽하게 유지만 하는 교착 상태 방지
- 캐주얼 게임 특성상 한 판이 너무 길어지면 이탈률 증가

### 2-9. 엣지케이스 처리

#### 연결 끊김

| 상황 | 처리 |
|------|------|
| 게임 중 한쪽 연결 끊김 | 30초 재접속 대기 |
| 30초 내 재접속 | 게임 이어서 진행 (끊긴 동안 유닛 자동 생산·전송 계속됨) |
| 30초 초과 | 상대방 몰수승 |
| 대기방에서 한쪽 나감 | 남은 사람 승리 처리 |
| 양쪽 모두 끊김 | 남은 사람 승리 / 둘 다 나가면 방 해제, 게임 무효 |

#### 예외 처리 (비정상 입력 방어)

| 상황 | 처리 |
|------|------|
| 상대 노드에서 엣지 생성 시도 | 서버에서 거부 (OwnerId 검증) |
| 자기 자신 노드로 연결 시도 | 서버에서 거부 |
| 노드당 연결 3개 초과 시도 | 서버에서 거부 (최대 연결 수 검증) |
| 동일 엣지 중복 생성 시도 | 서버에서 거부 |
| 유닛 0인 노드에서의 전송 | 자동 정지 (에러 아님, 정상 동작) |
| 게임 종료 후 조작 시도 | 서버에서 거부 (RoomStatus 검증) |
| 빠른 연타 클릭 | 서버 기준 상태로 처리 (클라이언트 수치 무시) |

## 3. 맵 설계

### 3-1. 맵 데이터 스키마 (JSON)

```json
{
  "mapId": "string",
  "mapName": "string",
  "description": "string",
  "width": 800,
  "height": 600,
  "unitSpeed": 150,
  "nodes": [
    {
      "id": "string (고유 식별자)",
      "name": "string (표시 이름)",
      "x": "number (0~width)",
      "y": "number (0~height)",
      "initialUnits": "number (중립 노드 초기 유닛 = 점령 문턱)",
      "maxUnits": "number (노드별 유닛 상한)",
      "isCapital": "boolean (향후 확장용)"
    }
  ],
  "spawnPoints": {
    "player1": "string (시작 nodeId)",
    "player2": "string (시작 nodeId)"
  },
  "playerStartUnits": 10
}
```

> **기존 대비 변경점:**
> - `edges` 필드 삭제 — 엣지는 맵에 고정되지 않고 플레이어가 직접 생성
> - `maxUnits` 필드 추가 — 노드별 유닛 상한 차등
> - `unitSpeed` 필드 추가 — 맵별 유닛 이동 속도

- Player1(서울)과 Player2(부산)은 남북 끝에서 시작, 초기 유닛·상한 대칭
- 대전/대구가 맵 중앙 거점 (초기 유닛 4, 상한 20으로 전략적 가치 높음)
- 서울↔부산 직선 거리 약 415px → 약 2.8초 (기습 가능하나 리스크 큼)
- 가까운 노드(인천, 청주 / 울산, 대구)를 먼저 확보하는 것이 유리
- 제주는 외곽(상한 10, 초기 2)이라 전략적 가치 낮지만 추가 생산기지로 활용 가능

### 3-4. 향후 맵 추가 가이드라인

새 맵을 추가할 때 지켜야 할 규칙:

1. **대칭성**: Player1과 Player2 시작 노드는 맵의 양 끝에 배치
2. **시작 유닛·상한 동일**: 양쪽 시작 노드의 initialUnits, maxUnits는 동일하게 설정
3. **거리 균등**: 양쪽에서 중앙 거점까지의 직선 거리가 유사하도록 설계
4. **노드 수**: 10~15개 권장
5. **상한 차등**: 거점(상한 높음) > 일반(중간) > 외곽(낮음) → 거점 쟁탈 유도
6. **초기 유닛 차등**: 거점은 높게(4~5), 외곽은 낮게(2~3) → 점령 난이도 차별화
7. **JSON만 추가하면 됨**: 로직 변경 없이 `Data/Maps/` 폴더에 JSON 파일 추가

## 4. 화면 상세 스펙

### 4-1. 화면 흐름

로비 → 대기 → 게임 → 결과 → 로비 (다시하기)

### 4-2. 로비 화면 (Lobby.razor)

**목적:** 게임 시작 전 방을 만들거나 참가

**UI 요소:**

| 요소 | 타입 | 설명 |
|------|------|------|
| 닉네임 입력 | TextInput | 최대 12자, 빈 값 불가 |
| 방 만들기 버튼 | Button | 닉네임 입력 후 클릭 시 방 코드 생성 → 대기 화면 이동 |
| 방 코드 입력 | TextInput | 숫자만 입력 (6자리) |
| 참가 버튼 | Button | 닉네임 + 코드 입력 후 방 입장 → 대기 화면 이동 |
| 에러 메시지 | Text | 방 없음 / 방 꽉참 / 닉네임 미입력 등 |

**상태 전환 조건:**

| 트리거 | 전환 |
|--------|------|
| 방 만들기 성공 | → 대기 화면 (방장) |
| 방 참가 성공 | → 대기 화면 (참가자) |
| 존재하지 않는 방 코드 | 에러 메시지 표시, 화면 유지 |
| 이미 2명인 방 | 에러 메시지 표시, 화면 유지 |
| 닉네임 미입력 | 에러 메시지 표시, 화면 유지 |

**동작 흐름:**
1. 닉네임 입력 (미입력 시 방 만들기/참가 버튼 비활성 또는 에러)
2. "방 만들기" → `CreateRoom` 이벤트 → 서버가 RoomId(숫자 6자리) 반환 → 대기 화면 이동
3. "참가" → `JoinRoom` 이벤트 → 성공 시 대기 화면 이동

### 4-3. 대기 화면 (Waiting.razor)

**목적:** 상대방 접속 대기 + 게임 시작 트리거

**UI 요소:**

| 요소 | 타입 | 설명 |
|------|------|------|
| 방 코드 | Text (굵게, 크게) | 상대에게 공유할 숫자 코드 |
| 맵 이름 | Text | 선택된 맵 표시 |
| Player1 닉네임 | Text | 방장 닉네임 표시 |
| Player2 닉네임 | Text | "대기 중..." 또는 참가자 닉네임 |
| 상태 텍스트 | Text | "상대를 기다리는 중..." / "곧 시작합니다!" |

**상태 전환 조건:**

| 트리거 | 전환 |
|--------|------|
| 2명 접속 완료 | 3초 카운트다운 → 게임 화면 |
| 한쪽 나감 | 남은 사람 승리 처리 |

**동작 흐름:**
1. 방장이 방 생성 후 대기
2. 참가자 입장 → `PlayerJoined` 이벤트 수신
3. 2명 완료 → 서버가 3초 카운트다운 시작
4. 카운트다운 완료 → `GameStarted` 이벤트 → 게임 화면 전환

### 4-4. 게임 화면 (Game.razor)

**목적:** 메인 게임 플레이

**레이아웃:**

> ┌──────────────────────────────────────────────────┐
> │  \[상단 HUD\]                                     │
> │  🔵 Player1(나): 45유닛 / 4노드                   │
> │  🔴 Player2:     38유닛 / 3노드    ⏱ 02:34       │
> ├──────────────────────────────────────────────────┤
> │                                                   │
> │                 \[SVG 맵 영역\]                    │
> │                                                   │
> │    ● 노드 + 이동 중인 유닛(작은 점)                │
> │    ● 클릭한 노드는 하이라이트 표시                  │
> │                                                   │
> ├──────────────────────────────────────────────────┤
> │  \[하단 상태바\]                                   │
> │  선택: 서울(12유닛) → 목표 노드를 클릭하세요        │
> └──────────────────────────────────────────────────┘

**UI 요소 상세:**

| 요소 | 설명 |
|------|------|
| **상단 HUD** | 양쪽 플레이어의 총 유닛 수 + 보유 노드 수, 게임 경과 시간 |
| **SVG 맵** | 노드(원) + 유닛 수(텍스트) + 이동 중인 유닛(작은 원) |
| **노드 색상** | 내 노드: `#3B82F6` (파랑), 상대 노드: `#EF4444` (빨강), 중립: `#9CA3AF` (회색) |
| **노드 선택 표시** | 선택된 노드: 외곽 링(glow) 하이라이트 |
| **이동 유닛 표시** | 연결된 경로를 따라 작은 원이 일정 속도로 이동 |
| **하단 상태바** | 현재 선택 상태 안내 텍스트 (미확정) |

> **엣지 시각화 옵션 (미확정, 테스트 후 결정):**
> - 옵션 A: 엣지 선 없이 이동 유닛(작은 점)만 표시 → 깔끔하지만 연결 상태 파악 어려움
> - 옵션 B: 얇은 점선으로 엣지 표시 + 이동 유닛 → 연결 상태 명확하지만 화면 복잡
> - 옵션 C: 내 연결만 표시, 상대 연결은 이동 유닛으로만 표시 → 절충안

**노드 인터랙션:**

| 동작 | 결과 |
|------|------|
| 내 노드 클릭 (비선택 상태) | 출발지 선택 (하이라이트 ON) |
| 선택 상태에서 다른 노드 클릭 | **연결 생성** → 유닛 자동 전송 시작 (`CreateEdge` 이벤트) |
| 선택 상태에서 이미 연결된 노드 클릭 | **연결 해제** → 전송 중단 (`RemoveEdge` 이벤트) |
| 선택 상태에서 같은 노드 다시 클릭 | 선택 해제 |
| 상대/중립 노드 클릭 (비선택 상태) | 아무 동작 없음 |
| 최대 연결(3개) 상태에서 새 연결 시도 | 에러 피드백 (진동/색상 등) |

**실시간 업데이트:**
- 서버에서 `GameStateUpdated` 수신 시 SVG 맵 리렌더링
- 이동 중인 유닛은 클라이언트에서 보간(interpolation) 처리하여 부드럽게 표시
- 유닛 전송/엣지 생성·해제는 즉시 서버로 전달, 서버 응답 기준으로 상태 반영

### 4-5. 결과 화면 (Result.razor)

**목적:** 게임 결과 확인 + 재시작/로비 복귀
