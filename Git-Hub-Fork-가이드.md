
# Fork & Pull Request 협업 가이드

---

## 이 방식이 뭔가요?

**Fork & PR**은 원본 저장소를 직접 건드리지 않고 협업하는 방법입니다.

팀원이 원본 레포를 **자기 계정으로 복사(Fork)** 해서 작업하고, 완성되면 **"이거 합쳐주세요(Pull Request)"** 라고 요청하는 흐름입니다.

> **왜 이렇게 하나요?**
> 여러 명이 하나의 레포에 직접 push하면 코드가 뒤섞이거나 실수가 바로 반영될 위험이 있습니다. Fork 방식은 레포 주인이 최종 검토 후 합치기 때문에 원본 코드를 안전하게 유지할 수 있습니다.

---

## 전체 흐름 한눈에 보기

> **[팀원]** 원본 레포 Fork → 내 컴퓨터에 Clone → 코드 작업 → Push → PR 생성 요청
>
> **[레포 주인]** PR 확인 → 코드 리뷰 → 문제 없으면 Merge

---

## 팀원이 할 일

### 1. 원본 레포 Fork (내 계정으로 복사하기)

1. 원본 레포지토리 페이지 우측 상단 **[Fork]** 버튼 클릭
2. 본인 GitHub 계정 선택 → **[Create fork]**
3. 이제 `내_계정/레포_이름` 형태로 내 계정에 복사본이 생깁니다

> ⚠️ Fork한 레포는 **내 계정의 복사본**입니다. 원본 레포와 별개입니다.

---

### 2. 내 컴퓨터로 Clone

Fork한 **내 계정의 레포**를 로컬로 내려받습니다.

```bash
git clone https://github.com/내_계정/레포지토리_이름.git
cd 레포지토리_이름
```

> ⚠️ 원본 레포 주소가 아니라 **내 계정(Fork된)** 레포 주소를 복사해야 합니다.

---

### 3. 코드 작업 후 Push

작업이 끝나면 변경 사항을 내 Fork 레포에 올립니다.

```bash
git add .
git commit -m "feat: 어떤 기능을 추가함"
git push origin main
```

---

### 4. Pull Request 생성

내 GitHub 레포 페이지에 들어가면 상단에 배너가 뜹니다.

1. **[Compare & pull request]** 버튼 클릭
2. 어떤 작업을 했는지 간단히 설명 작성
3. **[Create pull request]** 클릭 → 레포 주인에게 알림 전송

---

## 레포 주인이 할 일

### 5. PR 확인 & 코드 리뷰

1. 내 레포의 **[Pull requests]** 탭 클릭
2. 팀원이 보낸 PR 선택
3. **[Files changed]** 탭에서 어떤 코드가 바뀌었는지 확인
4. 코멘트 남기거나 수정 요청 가능

### 6. Merge

코드에 문제가 없다면:

**[Merge pull request]** → **[Confirm merge]**

이제 팀원의 코드가 원본 레포에 합쳐집니다. ✅

---

## 팁: 원본이 업데이트됐을 때 (Sync Fork)

내가 작업하는 동안 원본 레포에 다른 코드가 추가됐을 수 있습니다.
그 상태로 PR을 보내면 충돌(Conflict)이 날 수 있으니, 주기적으로 동기화해주세요.

**방법:** 내 Fork 레포 페이지 → **[Sync fork]** → **[Update branch]**

> 자주 동기화할수록 충돌 해결이 쉬워집니다.

## 브랜치 전략 방식과의 차이

| | **Fork & PR** | **브랜치 전략** |
|---|---|---|
| 대상 | 외부 참여자, 오픈소스 | 같은 팀, 동일 레포 권한 |
| 작업 공간 | 각자 Fork한 별도 레포 | 하나의 레포 안에서 브랜치 분리 |
| 원본 보호 | 레포 주인만 Merge 가능 | PR + 리뷰로 보호 |
| 복잡도 | 약간 번거로움 | 상대적으로 간단 |

> MintAndHeart 프로젝트처럼 팀원 모두 Collaborator로 등록된 경우엔 **브랜치 전략**이 더 적합합니다. Fork & PR은 외부 기여자를 받을 때 주로 사용합니다.
